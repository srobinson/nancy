# Emerging Configuration Formats Research (2024-2026)

**Research Date:** January 24, 2026
**Focus:** Alternative configuration formats competing with YAML/TOML
**Status:** Post-Pkl era analysis

---

## Executive Summary

The configuration language landscape in 2024-2026 shows **fragmentation without a clear winner**. While Apple's Pkl (Feb 2024) generated significant initial interest, no format has achieved breakthrough adoption. The market remains divided between:

1. **Production-stable legacy** (YAML, JSON, TOML) - still dominant
2. **Niche specialists** (CUE, Jsonnet, HOCON) - stable but contained
3. **New entrants** (Pkl, KCL) - hype vs. reality mismatch
4. **Academic/experimental** (Dhall, Nickel, RCL) - declining or hobby-tier

**Key Finding:** Organizations prioritize **ecosystem compatibility** over **language features**. The "perfect" configuration language exists in theory but struggles in practice due to tooling gaps, migration costs, and risk aversion.

## Detailed Analysis by Format

### 1. Pkl (Apple's Pickle)

**Official Site:** [pkl-lang.org](https://pkl-lang.org/)
**GitHub:** [apple/pkl](https://github.com/apple/pkl)
**Status:** New entrant (Feb 2024)

#### What Problem It Solves

- **Type-safe configuration:** Static typing with gradual type system
- **Rich validation:** Built-in constraints and validation rules
- **Multi-format output:** Generates JSON, YAML, TOML, Property Lists
- **IDE tooling:** First-class support for VS Code, IntelliJ, Neovim
- **Modular architecture:** Imports, functions, and reusable modules

#### Code Example

```pkl
module example.MyAppConfig

/// The hostname for the application
host: String

/// The port to listen on (must be > 1000)
port: UInt16(this > 1000)

/// Database configuration
database {
  url: String
  maxConnections: Int = 10
  timeout: Duration = 30.s
}

/// Collections
servers: Listing<String> = new {
  "web-1"
  "web-2"
  "web-3"
}
```

#### Adoption Metrics

- **GitHub Stars:** 10,800+ (rapid growth in 2024)
- **Release Cadence:** Active (v0.29.1 as of Aug 2025)
- **Language Bindings:** Go, Swift, Kotlin, Java
- **Industry Adoption:** Limited; mostly exploratory

#### Pros vs. TOML

- Rich type system with validation
- Programming constructs (functions, conditionals)
- Better tooling (LSP, formatters)
- Multi-format code generation
- Scales to complex configurations

#### Cons vs. TOML

- **Learning curve:** Steeper than TOML
- **Ecosystem maturity:** Brand new (2024)
- **Enterprise adoption risk:** Security teams skeptical of new languages
- **Migration cost:** Requires rewriting existing configs
- **Limited production examples:** Mostly Apple internal use

#### Momentum Assessment: **RISING (WITH CAVEATS)**

Pkl has the strongest 2024 launch of any config language, backed by Apple's brand and good engineering. However:

- **Question marks:** Why isn't Apple using it more publicly?
- **Competition:** CUE, KCL, and Dhall all tried similar approaches
- **Adoption lag:** Stars don't equal production deployments
- **Wait-and-see:** Most enterprises are watching, not adopting

### 2. CUE (Configure, Unify, Execute)

**Official Site:** [cuelang.org](https://cuelang.org/)
**GitHub:** [cue-lang/cue](https://github.com/cue-lang/cue)
**Status:** Mature (2018), Google-backed

#### What Problem It Solves

- **Types = Values:** Unique philosophy where types and values are unified
- **Constraint-based validation:** Define what configurations *must be*
- **Schema generation:** Converts to/from JSON Schema, OpenAPI, Protobuf
- **Data validation:** Validates YAML, JSON, TOML against constraints
- **Infrastructure as Code:** Popular for Kubernetes configs

#### Code Example

```cue
// Schema definition
#Server: {
    host!: string  // Required field
    port: int & >=1024 & <=65535 | *8080  // Default to 8080
}

// Validation constraints
area: length * width
area: <100  // Must be less than 100
width: 33.3 & >10
length: 5 & !=width  // Reject square areas

// Configuration with merge
development: #Server & {
    host: "localhost"
    port: 3000
}

production: #Server & {
    host: "api.example.com"
}
```

#### Adoption Metrics

- **GitHub Stars:** ~5,000-8,000 (estimated)
- **Production Users:** Infrastructure teams, Kubernetes shops
- **Notable Projects:** Timoni (Kubernetes package manager), dagger.io
- **Community:** Active with 24+ repositories in cue-lang org

#### Pros vs. TOML

- Powerful constraint system
- Schema reuse via unification
- Excellent for validating existing configs
- Strong Kubernetes ecosystem
- Incremental adoption path

#### Cons vs. TOML

- **Steep learning curve:** "Types are values" is mind-bending
- **Runtime validation only:** Performance bottlenecks in large configs
- **IDE support:** Less polished than Pkl
- **Documentation:** Conceptually dense
- **Niche philosophy:** Not everyone needs unification

#### Momentum Assessment: **STABLE**

CUE has found its niche in Kubernetes/cloud-native infrastructure. It's not growing explosively but has steady production adoption among SRE teams. The 2024-2025 period shows maintenance mode with no major breakthroughs.

### 3. KCL (Kubernetes Configuration Language)

**Official Site:** [kcl-lang.io](https://www.kcl-lang.io/)
**GitHub:** [kcl-lang/kcl](https://github.com/kcl-lang/kcl)
**Status:** CNCF Sandbox (Sept 2023)

#### What Problem It Solves

- **Kubernetes-native:** Purpose-built for K8s YAML generation
- **Policy enforcement:** Built-in validation and security policies
- **Configuration + Mutation + Validation:** Three-in-one approach
- **Cloud-native focus:** Integrations with Helm, Kustomize, ArgoCD
- **Gradual type system:** Static + dynamic typing like Python

#### Code Example

```kcl
import .app

# Basic Kubernetes app configuration
config = app.App {
    name = "kcl-guestbook-ui"
    containers.guestbook = {
        image = "gcr.io/heptio-images/ks-guestbook-demo:0.2"
        ports = [{containerPort = 80}]
    }
    service.ports = [{port = 80}]
    service.type = "LoadBalancer"
}

# Conditional annotation injection
schema Config:
    metadata?: {str:str} = {
        annotations?: {str:str}
    }

# Add annotations conditionally
items = [item | {
    if item.metadata?.annotations?["config.kubernetes.io/local-config"] == "true":
        metadata.annotations: {"configmanagement.gke.io/managed": "disabled"}
} for item in option("items")]
```

#### Adoption Metrics

- **GitHub Stars:** ~3,000-5,000 (estimated)
- **CNCF Status:** Sandbox project (Sept 2023)
- **Integrations:** Kubectl, Kustomize, Helm, ArgoCD, Crossplane, Flux
- **Release Cadence:** Biweekly newsletters through 2024
- **2024 Roadmap:** Targeting CNCF incubation (ambitious)

#### Pros vs. TOML

- Purpose-built for Kubernetes
- CNCF backing (credibility)
- Rich policy/validation features
- Python-like syntax (familiar)
- Active development (newsletters, releases)

#### Cons vs. TOML

- **K8s-only focus:** Not general-purpose
- **Sandbox maturity:** Not yet incubating/graduated
- **Adoption uncertainty:** CNCF Sandbox ≠ production readiness
- **Ecosystem lock-in:** Heavily CNCF-centric
- **Compared to Pkl:** Less general, more domain-specific

#### Momentum Assessment: **RISING (KUBERNETES NICHE)**

KCL is the most credible YAML alternative *specifically for Kubernetes*. CNCF backing matters. However:

- Sandbox status is early-stage (most projects never graduate)
- Production adoption metrics not disclosed
- Competes with Helm/Kustomize inertia
- 2024 goal of incubation appears missed (as of Jan 2026)

### 4. Dhall

**Official Site:** [dhall-lang.org](https://dhall-lang.org/)
**GitHub:** [dhall-lang/dhall-lang](https://github.com/dhall-lang/dhall-lang)
**Status:** Mature but declining

#### What Problem It Solves

- **Total functions:** Guaranteed termination (not Turing-complete)
- **Programmable configs:** Functions, imports, type safety
- **Haskell integration:** First-class citizen in Haskell ecosystem
- **Security:** Sandboxed imports with integrity checks

#### Code Example

```dhall
-- Simple configuration
{ name = "example"
, port = 8080
, database =
    { host = "localhost"
    , user = "admin"
    }
}

-- With functions and types
let Config =
    { Type =
        { name : Text
        , port : Natural
        }
    , default =
        { name = "app"
        , port = 3000
        }
    }

in Config::{
    name = "my-service"
}
```

#### Adoption Metrics

- **GitHub Stars:** 4,393 (main repo)
- **Language Bindings:** Haskell, Rust, Python, Go
- **Community:** Mature but shrinking

#### Pros vs. TOML

- Strong type system
- Guaranteed termination (safer than Turing-complete)
- Functions and imports
- Excellent documentation

#### Cons vs. TOML

- **Declining adoption:** Projects moving *away* from Dhall
- **Example:** Spago (PureScript package manager) deprecated Dhall for YAML
- **Rust maintainer abandoned it:** "Lost faith in usability"
- **Haskell-centric:** Less relevant outside that ecosystem
- **Over-engineered:** Total function guarantee is academic overkill for configs

#### Momentum Assessment: **DECLINING**

Dhall peaked around 2018-2020. By 2024-2025, key projects are migrating away:

- Spago switching to YAML is a red flag
- Maintainers citing "usability" concerns
- Haskell ecosystem itself is niche
- No new major adopters

Dhall is technically sound but failed the adoption test. It's the cautionary tale for config languages: **perfect features don't guarantee success**.

### 5. Nickel

**Official Site:** [nickel-lang.org](https://nickel-lang.org/)
**GitHub:** [nickel-lang/nickel](https://github.com/tweag/nickel)
**Status:** Stable 1.0 (May 2023)

#### What Problem It Solves

- **JSON with functions:** Gradual type system, superset of JSON
- **Merge-oriented:** Composing configs from multiple sources
- **Contract system:** Mix static and dynamic validation
- **Nix-inspired:** Targets infrastructure-as-code use cases
- **Tweag-backed:** Professional engineering from Modus Create

#### Code Example

```nickel
# Simple configuration
{
  name = "example",
  description = m%"
    This is an awesome software I'm developing.
    Multiline strings are awesome!
  "%,
  version = "1.0.0",

  # Nested objects
  server = {
    host = "localhost",
    port = 8080,
  },

  # With contracts (validation)
  age | Number
      | std.number.Positive
    = 25,
}
```

#### Adoption Metrics

- **GitHub Stars:** ~2,000-3,000 (estimated)
- **Maturity:** 1.0 released May 2023, now at 1.12.2 (June 2025)
- **Tooling:** LSP, REPL, doc generator
- **Adoption:** Tweag's own Topiary project switched from TOML to Nickel (2024)

#### Pros vs. TOML

- Gradual typing (optional)
- Functions and imports
- JSON compatibility
- Contract system for validation
- Good documentation

#### Cons vs. TOML

- **Niche adoption:** Mostly Tweag internal
- **Nix ecosystem focus:** Limited general-purpose traction
- **Competition:** Overlaps with Dhall, CUE, Pkl
- **Risk:** Small community, vendor-dependent

#### Momentum Assessment: **STABLE (NICHE)**

Nickel reached 1.0 in 2023 and has been actively maintained through 2025. However:

- Adoption is mostly within Tweag/Nix community
- Topiary adoption (2024) shows confidence but small scale
- No breakout production examples
- Technically solid but lacks ecosystem momentum

### 6. Jsonnet

**Official Site:** [jsonnet.org](https://jsonnet.org/)
**GitHub:** [google/jsonnet](https://github.com/google/jsonnet)
**Status:** Mature, stable adoption

#### What Problem It Solves

- **JSON templating:** Functions, variables, imports for DRY configs
- **Kubernetes focus:** Widely used for K8s YAML generation
- **Data transformation:** Generate JSON from logic
- **Library ecosystem:** Extensive Kubernetes libraries

#### Code Example

```jsonnet
// Simple template
{
  name: "example",
  port: 8080,
  replicas: 3,

  // Computed values
  instances: [
    "instance-%d" % i
    for i in std.range(1, $.replicas)
  ],

  // Functions
  local makeContainer(name, image) = {
    name: name,
    image: image,
    ports: [{containerPort: $.port}],
  },

  containers: [
    makeContainer("web", "nginx:latest"),
  ],
}
```

#### Adoption Metrics

- **GitHub Stars:** 7,300
- **Production Use:** Databricks (40,000+ lines), Grafana Tanka
- **Kubernetes Libraries:** jsonnet-libs/k8s-libsonnet (active through 2024-2026)
- **Age:** Released 2015, still maintained

#### Pros vs. TOML

- Proven production track record
- Large Kubernetes ecosystem
- Functions and templating
- Good documentation

#### Cons vs. TOML

- **Aging design:** 2015 syntax feels dated vs. Pkl/Nickel
- **No type system:** Runtime errors only
- **Kubernetes-centric:** Limited general-purpose adoption
- **Google maintenance:** Unclear long-term commitment

#### Momentum Assessment: **STABLE (LEGACY)**

Jsonnet is the *incumbent* for Kubernetes templating. It's not sexy or new, but it works:

- Still actively maintained (k8s libs updated for K8s 1.33 in 2024-2025)
- Production-proven at scale (Databricks)
- Not growing fast, but not declining either
- Competes with Helm/Kustomize, not TOML/YAML

**Sources:**

- [Jsonnet GitHub Repository](https://github.com/google/jsonnet)

### 7. HOCON (Human-Optimized Config Object Notation)

**Official Site:** [GitHub - lightbend/config](https://github.com/lightbend/config)
**Spec:** [HOCON Specification](https://github.com/lightbend/config/blob/main/HOCON.md)
**Status:** Stable, JVM niche

#### What Problem It Solves

- **JSON superset:** Adds comments, includes, substitutions
- **JVM integration:** Native support in Akka, Play Framework
- **Human-friendly:** More readable than JSON
- **Configuration merging:** Layered configs with overrides

#### Code Example

```hocon
# HOCON configuration
app {
  name = "MyApp"
  version = "1.0"

  # Substitutions
  database {
    url = "jdbc:postgresql://localhost/mydb"
    user = ${DB_USER}
    password = ${?DB_PASSWORD}  # Optional
  }

  # Includes
  include "common.conf"

  # Lists and objects
  servers = [
    {host: "server1", port: 8080}
    {host: "server2", port: 8081}
  ]
}
```

#### Adoption Metrics

- **GitHub Stars:** N/A (part of Typesafe Config library)
- **Ecosystem:** Akka, Play, Sponge, many JVM projects
- **Maturity:** ~12 years old (since ~2012)
- **Status:** "Feature complete" per Lightbend

#### Pros vs. TOML

- More flexible syntax
- Variable substitution
- Include files
- JVM ecosystem standard

#### Cons vs. TOML

- **JVM-only:** Rarely used outside Scala/Java
- **Niche:** Not competing with YAML/TOML in general use
- **Feature complete = maintenance mode:** No new development
- **2024-2026:** No significant changes or growth

#### Momentum Assessment: **STABLE (NICHE)**

HOCON is the JVM's answer to configuration. It's not growing, not shrinking, just *there*:

- Perfect for Akka/Scala users
- Irrelevant outside JVM
- No plans to expand beyond niche
- 2024-2026: business as usual

### 8. RCL (Reasonable Configuration Language)

**Official Site:** [rcl-lang.org](https://rcl-lang.org/)
**GitHub:** [ruuda/rcl](https://github.com/ruuda/rcl)

#### Code Example

```rcl

  name = "example",
  port = 8_080,  // Numeric underscores

  servers = [
    "web-1",
    "web-2",  // Trailing comma allowed
  ],

  // Functions
  make_url(host: String) = "https://" + host,

  urls = [make_url(s) for s in servers],
}
```

#### Adoption Metrics

- **GitHub Stars:** <500 (estimated)
- **Maturity:** Experimental, no stability promise
- **Adoption:** Hobby project, author ambivalent about growth

#### Pros vs. TOML

- Good error messages
- Exports to JSON/YAML/TOML
- Well-documented

#### Cons vs. TOML

- **Hobby project:** No organizational backing
- **Author's quote:** "I'm not sure I want it to experience prime time" (maintainer fatigue)
- **No ecosystem:** No tooling, no libraries
- **2024 release:** Too new to judge viability

#### Momentum Assessment: **EXPERIMENTAL**

RCL is a well-executed hobby project that may never escape that label:

- Author is explicit about lack of commitment
- No syntax highlighting on GitHub (visibility issue)
- Technically interesting but organizationally DOA
- Unlikely to challenge TOML/YAML

## Key Questions Answered

### Is there a "next generation" config format emerging?

**Short answer:** No clear winner yet.

**Analysis:**

- **Pkl** has the strongest 2024 launch (Apple branding, good engineering, 10K stars)
- **KCL** has CNCF credibility for Kubernetes-specific use cases
- **CUE** holds steady as the "thinking person's" choice for infrastructure

However, none have achieved breakthrough adoption. Reasons:

1. **Migration inertia:** Existing YAML/TOML codebases are "good enough"
2. **Tooling gaps:** Even Pkl (best-in-class) lacks ecosystem breadth of YAML
3. **Risk aversion:** Enterprises don't adopt new languages lightly
4. **Format fatigue:** "Not another config language" is a real sentiment

### What capabilities do they offer beyond YAML/TOML?

**Core capabilities:**

1. **Type safety:** Catch errors before runtime (Pkl, CUE, Dhall, KCL)
2. **Validation:** Built-in constraints and policy enforcement
3. **Functions:** DRY configurations via reusable logic
4. **Imports/modules:** Compose configs from multiple files
5. **IDE support:** Autocomplete, type checking, refactoring
6. **Multi-format output:** Write once, generate JSON/YAML/TOML

**Why these matter:**

- **Large-scale infrastructure:** Managing 1000+ Kubernetes manifests
- **Policy enforcement:** Security teams want validation-as-code
- **Developer experience:** Autocomplete >> reading docs

**Why they don't matter (yet):**

- Most projects are small/medium scale
- YAML + validation tools (kubeval, ajv) work "well enough"
- Learning curves deter casual users

### Are any gaining serious production adoption?

**Production-proven (but niche):**

- **Jsonnet:** Databricks (40,000 lines), Grafana ecosystem
- **CUE:** Infrastructure teams at cloud-native shops
- **HOCON:** Akka/Play Framework (JVM ecosystem standard)

**Exploratory adoption:**

- **Pkl:** Early adopters experimenting (2024-2025)
- **KCL:** CNCF ecosystem testing (Helm/Kustomize integrations)

**Not yet production-ready:**

- **Nickel:** Mostly Tweag internal
- **Dhall:** Declining (projects migrating away)
- **RCL:** Hobby tier

**Key insight:** "GitHub stars ≠ production deployments"

Pkl's 10,800 stars are impressive, but there's scant evidence of real-world production use outside Apple. Compare to Jsonnet's 7,300 stars but proven Databricks deployment.

### Which ones are abandoned/stalled?

**Declining:**
- **Dhall:** Maintainers losing faith, projects migrating away (Spago → YAML)

**Stalled/Maintenance Mode:**
- **HOCON:** "Feature complete" = no new development

**At Risk:**
- **RCL:** Author explicit about not wanting growth
- **Nickel:** Small community, Tweag-dependent

**Still Active:**
- **Pkl, KCL, CUE, Jsonnet:** All show 2024-2025 activity

## Pros/Cons vs. TOML

### When Alternative Formats Win

**Use alternative formats when:**

1. **Large-scale infrastructure:** Managing 100+ config files
2. **Type safety required:** Compliance/security mandates
3. **Complex validation:** Policy-as-code enforcement
4. **Team expertise:** Team already knows the format
5. **Ecosystem lock-in acceptable:** K8s-only (KCL), JVM-only (HOCON)

**Examples:**
- CUE for Kubernetes infrastructure at cloud-native companies
- Pkl for Apple internal projects (once they actually adopt it...)
- KCL for CNCF-heavy shops with GitOps workflows
- Jsonnet for Grafana/Databricks-style K8s management

### When TOML/YAML Still Win

**Stick with TOML/YAML when:**

1. **Simplicity matters:** Small projects, single files
2. **Universal tooling:** Every language has YAML/TOML parsers
3. **Human-editable:** Non-programmers need to edit configs
4. **No validation needs:** Configuration is straightforward
5. **Risk aversion:** Can't justify learning curve or migration cost

**Examples:**
- `pyproject.toml` for Python projects
- `Cargo.toml` for Rust
- GitHub Actions YAML
- Docker Compose YAML

### The Uncomfortable Truth

**Most projects don't need more than YAML/TOML.**

The advanced features (functions, types, validation) solve real problems, but only at scale. For the median project:

- **YAML + JSON Schema validation** is sufficient
- **TOML + basic linting** handles 80% of use cases
- **Migration cost >> benefit** for most teams

Alternative formats are solutions looking for problems in the median case, but genuine lifesavers in the 90th percentile case (large infrastructure).

## The Bigger Picture: Why Hasn't a Winner Emerged?

### Thesis: The Configuration Language Paradox

**Good configuration languages optimize for:**
1. Type safety
2. Validation
3. Abstraction (DRY)
4. IDE support

**But successful configuration formats optimize for:**
1. Simplicity
2. Universal tooling
3. Human editability
4. Ecosystem adoption

**These goals are often in conflict.**

### Evidence from 2024-2026

1. **Pkl's launch (2024):** Technically excellent, but adoption muted
   - Great IDE support, types, validation
   - But: learning curve, migration cost, risk

2. **Dhall's decline:** Over-engineering kills adoption
   - Total functions (guaranteed termination) is academically beautiful
   - But: Spago switched to YAML because "simpler"

3. **Jsonnet's persistence:** "Good enough" beats "perfect"
   - No types, no safety, aging syntax
   - But: works at Databricks scale, proven, stable

4. **YAML's dominance:** Worse is better
   - Everyone complains about YAML (indentation, Norway problem, etc.)
   - But: every tool supports it, everyone knows it

### The Adoption Equation

```
Adoption = (Feature_Value - Learning_Curve - Migration_Cost) / Ecosystem_Lock_In
```

**Why Pkl hasn't won (yet):**
- High `Feature_Value` (types, IDE, validation)
- Medium `Learning_Curve` (steeper than TOML)
- High `Migration_Cost` (rewrite existing configs)
- Low `Ecosystem_Lock_In` (new, few integrations)

**Result:** Wait-and-see adoption

**Why TOML persists:**
- Medium `Feature_Value` (good enough)
- Low `Learning_Curve` (5 minutes)
- Zero `Migration_Cost` (for new projects)
- High `Ecosystem_Lock_In` (Rust, Python standards)

**Result:** Default choice

## Conclusion

### The State of Config Formats (2024-2026)

**No revolution, just evolution.**

The quest for the "perfect" configuration language continues, but 2024-2026 shows that **adoption is harder than engineering**. Pkl is the best-engineered config language ever released, yet its impact is TBD. KCL has CNCF backing but Sandbox status. CUE has found a niche but not dominance. Dhall is declining despite technical excellence.

Meanwhile, YAML and TOML persist because:
- **Simplicity wins** for most use cases
- **Ecosystem lock-in** is powerful (Rust = TOML, K8s = YAML)
- **Migration cost** is underestimated by format designers

### Key Takeaways

1. **For most projects:** Stick with TOML/YAML + validation tooling
2. **For Kubernetes at scale:** Consider CUE or Jsonnet (proven) or bet on KCL (emerging)
3. **For new Apple-ecosystem projects:** Experiment with Pkl
4. **For JVM projects:** HOCON is fine, don't change
5. **Avoid:** Dhall (declining), RCL (hobby), over-engineering simple configs

### The Uncomfortable Question

**Do we even need new configuration languages?**

The 2024-2026 data suggests the answer is **"not for most people."** The formats we have—YAML, TOML, JSON—handle 80% of use cases adequately. The remaining 20% (large-scale infrastructure) has niche solutions (CUE, Jsonnet) that work.

New formats like Pkl and KCL are technically impressive but face a high bar:
- Better enough to justify rewriting existing configs?
- Better enough to justify learning curves?
- Better enough to justify ecosystem fragmentation?

For most teams, the answer is "not yet."

### Final Verdict

**Emerging formats (2024-2026) are solutions in search of problems at the median scale, but essential tools at the 95th percentile scale.**

Choose wisely based on your actual needs, not theoretical elegance.
