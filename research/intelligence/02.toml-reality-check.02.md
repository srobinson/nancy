# Configuration Format Reality Check: 2025-2026 Tools Analysis

**Research Date:** January 24, 2026
**Focus:** Tools launched or significantly updated in 2025-2026
**Objective:** Document configuration format choices and their reasoning

---

## Executive Summary

### Key Findings

1. **JSON Remains Dominant** for AI/LLM tools and extensions, primarily due to VS Code ecosystem compatibility
2. **YAML Gaining Ground** in AI agent frameworks and multi-agent systems (CrewAI, Continue.dev)
3. **TOML Strong in Python/Rust Ecosystems** (uv, Ruff, mise, pnpm v10+)
4. **Format Migrations Happening** - Several major tools transitioned formats in 2025:
   - Continue.dev: JSON → YAML
   - pnpm v10: INI → JSON/camelCase
5. **Developer Experience Priority** - Configuration simplification is a major trend across all categories

### Model Context Protocol (MCP)
- **Specification:** November 25, 2025 (one-year anniversary release)

- **Why Multiple Formats:** Client-dependent (Codex uses TOML, others use JSON)
- **Recent Update:** June 2025 - Authorization handling clarifications

**Example (TOML):**

## Category 3: Package Managers & Runtime Tools

### uv (Astral)
- **Status:** Active development (2025-2026)
- **Format:** TOML (pyproject.toml with `[tool.uv]`, or uv.toml)
- **Configuration Hierarchy:**
  - `uv.toml` (takes precedence)
  - `pyproject.toml` with `[tool.uv]` prefix
  - User-level and system-level configs
- **Why TOML:** Python ecosystem standard (PEP 621), aligns with Rust/Cargo
- **Notable:** uv.toml omits `[tool.uv]` prefix for cleaner syntax

**Quote:** "uv will search for a pyproject.toml or uv.toml file in the current directory, or in the nearest parent directory."

**Sources:**
- [uv Configuration Files](https://docs.astral.sh/uv/concepts/configuration-files/)
- [uv Working on Projects](https://docs.astral.sh/uv/guides/projects/)

### mise (mise-en-place)
- **Latest:** mise 2026.1.6 (Jan 21, 2026)
- **Format:** TOML (mise.toml, .miserc.toml)
- **Major Updates:**
  - New `.miserc.toml` for early initialization
  - Enhanced error diagnostics with miette
  - Deprecated Tera template functions (removal in 2026.11.0)
- **Why TOML:** Developer tool ecosystem preference, readability
- **Configuration Hierarchy:** mise.toml → mise.local.toml (for local overrides)

**Quote:** "When you make a typo in your mise.toml, you'll now get beautifully formatted error messages with precise line and column information."

**Sources:**
- [mise Configuration](https://mise.jdx.dev/configuration.html)
- [mise Releases](https://github.com/jdx/mise/releases)

### pnpm
- **Major Release:** pnpm v10 (2025)
- **Format:** YAML (pnpm-workspace.yaml), camelCase JSON for config output
- **Breaking Changes:**
  - Config output: INI → JSON
  - `pnpm config list` now outputs JSON object
  - Package-specific configs moved to `packageConfigs` in pnpm-workspace.yaml
  - No longer loads non-auth/non-registry settings from rc files
- **Why Changes:** Consistency, programmatic access, security
- **New Features:**
  - `allowBuilds` replacing `onlyBuiltDependencies`
  - `minimumReleaseAge` (zero-day protection)

**Quote:** "pnpm v10.26 introduced `allowBuilds` to replace `onlyBuiltDependencies`, offering more flexible control over which packages can run build scripts."

**Sources:**
- [pnpm in 2025](https://pnpm.io/blog/2025/12/29/pnpm-in-2025)
- [pnpm Settings](https://pnpm.io/settings)

### Bun
- **Latest:** v1.3.6 (Jan 13, 2026)
- **Format:** TOML (bunfig.toml), TypeScript (bunup.config.ts)
- **Major Updates v1.3:**
  - Zero-config frontend development
  - Bun.SQL unified API
  - Control over .env and bunfig.toml loading in standalone executables
- **Why Multiple Formats:**
  - TOML for runtime config (aligns with .env philosophy)
  - TypeScript for build configs (type safety, IDE support)
- **Reception:** "Biggest release yet" - full-stack capabilities

**Sources:**
- [Bun Releases](https://github.com/oven-sh/bun/releases)
- [Bun v3.1 Release](https://www.infoq.com/news/2026/01/bun-v3-1-release/)

## Analysis: Format Distribution & Trends

### By Category

**AI/LLM Tools (8 tools):**
- JSON: 6 (75%) - Cursor, Windsurf, Zed, Roo Code, Cody (IDE), Warp
- YAML: 2 (25%) - Aider, Continue.dev (migrated)

**AI Agent Frameworks (5 frameworks):**
- YAML: 2 (40%) - CrewAI, MCP (some clients)
- Code-first: 3 (60%) - LangChain/LangGraph, OpenAI SDK, Google ADK
- TOML: 1 (20%) - MCP (Codex)

**Package Managers (4 tools):**
- TOML: 3 (75%) - uv, mise, Bun
- YAML: 1 (25%) - pnpm (workspace config)
- JSON: 1 (25%) - pnpm (config output)

**Build Tools (5 tools):**
- JSON: 3 (60%) - Turborepo, Biome, Oxfmt
- TOML: 1 (20%) - Ruff
- JS/TS: 1 (20%) - Vite

**Web Frameworks (4 frameworks):**
- JS/TS: 4 (100%) - Next.js, Astro, SvelteKit, Remix

**Infrastructure (2 tools):**
- YAML: 1 (50%) - Temporal
- JSON: 1 (50%) - Temporal
- Code-first: 1 (50%) - Dagger

### Overall Trends

#### 1. VS Code Ecosystem = JSON Lock-in
Every AI coding assistant uses JSON for IDE integration. No exceptions.

**Why:** Extension API, settings.json standard, ecosystem compatibility

#### 2. YAML Rising in Multi-Agent Systems
CrewAI, Continue.dev, Aider all chose YAML for complex, hierarchical configs.

**Why:** Human readability, nested structures, comments support

#### 3. TOML Dominance in Python/Rust Tools
uv, Ruff, mise, Bun, MCP (Codex) - all modern developer tools.

**Why:** "Minimalist and easy-to-read" - balance between conciseness and structure

#### 4. Code-First for Maximum Flexibility
LangChain, Google ADK, OpenAI SDK, Dagger (post-CUE).

**Why:** Type safety, programmatic control, no file parsing overhead

#### 5. Format Migrations Happening
- **Continue.dev:** JSON → YAML (2025)
- **pnpm:** INI → JSON (v10, 2025)
- **Dagger:** CUE → Code-first (2023-2024)

**Why:** Developer experience improvements, ecosystem alignment

## Migration Stories

### Continue.dev: JSON → YAML (2025)
**Reason:** "Building custom coding agents" required more expressive configuration
**Impact:** Better support for complex agent definitions, rules, and MCP servers
**Reception:** Positive - aligns with CrewAI and other agent frameworks

## Recommendations Based on 2025-2026 Data

### Choose JSON If:
- Building VS Code extension or IDE integration
- Need universal compatibility across languages
- API data transport or config-as-API response
- Strict validation via JSON Schema required
- TypeScript type generation from config

**Modern Examples:** Cursor, Windsurf, Zed, Roo Code, Biome, Turborepo

### Choose YAML If:
- Complex, hierarchical configuration (multi-agent systems)
- Human readability is paramount (documentation, DevOps)
- Comments and documentation within config essential
- Kubernetes/cloud-native ecosystem integration
- Multi-environment configs with anchors/references

**Modern Examples:** CrewAI, Continue.dev, Aider, pnpm-workspace, Temporal

### Choose TOML If:
- Developer tool with config files (not data transport)
- Python or Rust ecosystem integration
- Want balance: readable but less error-prone than YAML
- Date/time types needed natively
- Explicit key-value pairs over nested structures

**Modern Examples:** uv, Ruff, mise, Bun, MCP (Codex)

### Choose Code-First If:
- Maximum flexibility and programmatic control required
- Type safety across configuration essential
- Config complexity exceeds file format capabilities
- Runtime decision-making in config needed
- Building framework/SDK (not end-user tool)

**Modern Examples:** LangChain/LangGraph, OpenAI SDK, Google ADK, Dagger, Next.js

## Key Takeaways

1. **No Format "Won"** - Context determines choice
2. **JSON Still King** for IDE/extension ecosystem
3. **YAML Gaining** in AI agent frameworks (human + machine readable)
4. **TOML Strong** in modern dev tools (Rust/Python influence)
5. **Code-First Growing** for frameworks requiring maximum flexibility
6. **Migrations Happen** - Plan for them (Continue, pnpm, Dagger)
7. **DX Matters Most** - Format choice less important than error messages, validation, migration tools

### The Real Trend: Configuration Minimization

> "The era of choosing a router or configuring a bundler is largely over."

Tools are moving toward:
- Zero-config defaults
- Convention over configuration
- AI-assisted config generation
- Automatic migration between versions

**Format choice matters, but simplicity matters more.**
